# Decision Flow - 決策流程

## 任務處理流程

### 1️⃣ 任務接收
**JARVIS** 接收用戶指令，進行初步分析。

### 2️⃣ 任務分解
**JARVIS** 將任務分解，分配給相關子 agent。

### 3️⃣ 子 Agent 討論
相關子 agent 根據各自專長提供意見：
- **ATHENA** → 資料分析、事實驗證
- **BLAZE** → 創意方案、突破性想法
- **SENTINEL** → 風險評估、安全檢查
- **NEXUS** → 架構設計、整合方案
- **ECHO** → 用戶視角、溝通建議

### 4️⃣ 意見整合
**JARVIS** 整合所有意見，權衡利弊。

### 5️⃣ 最終決策
**JARVIS** 做出最終決策，可能：
- ✅ 直接執行（日常任務）
- 📋 向用戶回報並等待確認（重大決策）
- 🔄 要求子 agent 進一步分析

### 6️⃣ 執行與回報
執行決策，向用戶回報結果。

---

## 決策權限

| 決策類型 | 權限 | 需用戶確認 |
|----------|------|------------|
| 日常任務執行 | JARVIS | ❌ |
| 新技能安裝 | JARVIS | ❌ |
| 代碼推送 | JARVIS | ❌ |
| 外部 API 調用 | JARVIS | ⚠️ 視情況 |
| 刪除/破壞性操作 | 用戶 | ✅ |
| 專案方向變更 | 用戶 | ✅ |
| 敏感資料處理 | 用戶 | ✅ |

---

## 會議記錄格式

當子 agent 討論時，使用以下格式：

```
---
**ATHENA:** [分析內容]
**BLAZE:** [創意建議]
**SENTINEL:** [風險警告]
**NEXUS:** [架構建議]
**ECHO:** [用戶視角]

**JARVIS 決策:** [最終決定]
---
```

---

## 緊急情況

如遇緊急或高風險情況，**SENTINEL** 有權直接要求暫停並通知用戶。
